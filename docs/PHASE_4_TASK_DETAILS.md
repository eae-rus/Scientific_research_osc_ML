## Q1: Обработка отсутствующих сигналов (NaN) {#q1}

**Суть проблемы:** Когда физически отсутствует канал (например, ток фазы B), модель может воспринимать это как аномалию (нулевой ток = авария), что некорректно.

**Что нужно проверить:**
- Корректность маркировки: отсутствующий канал → NaN на всём пути (датасет → stem → attention mask → loss).
- Что `DataSanitizer` (замена NaN → 0 + learnable MissingToken + binary mask) действительно не создаёт «ложного» паттерна для модели (если оно работает так).
- Что `key_padding_mask` в Attention корректно исключает NaN-позиции.

**Контекст:** Вопрос уже частично разбирался ранее (возможно, есть записи в PHASE_4_WORK_LOG.md). В текущей реализации: `dataset_manager.py` детектирует `missing_ch` → NaN; `DataSanitizer` → -1→0 + mask; `ComplexMSELoss` принимает mask.

### Результат верификации (2026-03-31) — РАСШИРЕНО

**Полная цепочка прохождения mask:**

1. **`DataSanitizer`** (transformer_blocks.py) — ✅ Корректен:
   - `torch.isnan(x)` → `mask_missing` shape `(B, C, T)` 
   - `torch.where(mask_missing, 0, x)` → безопасные данные
   - `x_safe + mask_missing.float() * missing_token` → learnable embedding
   - Возвращает `(x_safe, mask_missing)`

2. **`PhysicalStem.forward`** — ⚠️ **Принимает `mask_missing`, но НЕ использует его!**
   - Параметр принят, но нигде в теле не упоминается
   - Пропущенные данные (0 + missing_token) проходят через KAN, interaction и проекции без маскировки
   - Не критично (NaN уже заменены на 0), но теоретически некорректно: KAN может «выучить» паттерн 0+token
   - **TODO:** Рассмотреть маскирование вычислений в Stem для позиций с `mask_missing.all(dim=1)`

3. **`key_padding_mask` в Attention** — ✅ Корректен:
   - `all_missing_mask = mask_missing.all(dim=1)` — True только если ВСЕ каналы на шаге T пропущены
   - Передаётся в `nn.MultiheadAttention` → маскированные позиции получают `-inf` в softmax

4. **Проблема NaN при полной маскировке** — ✅ Защита добавлена:
   - Если **все** позиции в окне замаскированы, `softmax(-inf, ..., -inf) = NaN`
   - Добавлен `torch.nan_to_num(attn_output, nan=0.0)` в `TransformerEncoderBlock.forward()`
   - На практике маловероятно (хотя бы 1 канал обычно есть), но защита страхует

**Статус:** Верифицировано. Два TODO остаются:
- Использование mask в PhysicalStem (низкий приоритет, не влияет на корректность)
- Тестирование на реальных данных с обширными пропусками (отдельный эксперимент)

## Q3: Проверка FUTURE_PERIODS в разных режимах {#q3}

**Суть:** Нужно убедиться, что при `FUTURE_PERIODS > 0`:
- В режиме **zone** — модель предсказывает будущее (это нормально и полезно).
- В режиме **last_point** — модель предсказывает значение **последней** точки текущего окна (10 периодов), без сдвига на будущие периоды, даже если на претрейне училась этому.
- В режиме **window** — аналогично, предсказание по текущему окну, без учёта будущих периодов.

**Мотивация:** Если модель в режиме last_point пытается предсказать именно «будущее», а не текущее состояние, это приводит к некорректной интерпретации результатов и существенно большей ошибке. Да, предсказывающая модель будет, и актуальна, но только если процесс уже движется.

### Результат верификации (2026-03-31) — ИСПРАВЛЕНО

**Обнаружена ошибка:** В `_format_classify` использовалось `T_use = num_steps_full` (включая future),
то есть модель **видела FFT-признаки будущих периодов в X**. Это некорректно для задачи РЗА — 
модель не должна «заглядывать в будущее», она должна **предсказывать** будущее только по текущему контексту.

**Исправление (2026-03-31):** `_format_classify` теперь использует `T_use = num_steps_current`
(только текущие 10 периодов). Y также обрезается до `num_steps_current` зон.
Будущие периоды остаются исключительно в SSL pretrain (task: future prediction).

**Реализованная архитектура (2026-03-31):**

1. **`_format_classify`**: X = `num_steps_current` (модель НЕ видит будущие FFT), Y = `num_steps_full` (текущие + будущие зоны)
2. **`FuturePredictionHead`** (авторегрессионная голова, `osc_tools/ml/models/transformer.py`):
   - Обучаемые future_queries (по одному на будущую зону, как в DETR)
   - Cross-attention: queries обращаются к hidden states encoder (только текущие данные)
   - Каждый query предсказывает метку своей будущей зоны
   - Выход конкатенируется с зональными предсказаниями: `[cls_current | future_pred]`
3. **`_reduce_logits_targets`**: защитное выравнивание `T_logits != T_targets` → обрезка до min

**Смысл для режимов:**
- **zone:** модель предсказывает текущие зоны по данным + future зоны через авторегрессию (не видит будущего)
- **window / last_zone:** агрегация по всем зонам (включая future predictions) — логистика max/mean
- **Если нужно расширенное окно** — увеличиваем `NUM_FULL_PERIODS`, а не используем future
- **Предсказание будущего** — реализовано через FuturePredictionHead с cross-attention

## Q4: Отключение физических блоков {#q4}

**Цель:** Сделать простым и быстрым отключение блоков деления, умножения, цепи управления амплитудой через угол. Это нужно для двух типов исследований:

1. **Inference-абляция:** На уже обученной модели отключить/занулить выходы физ. блоков → измерить падение точности → определить наиболее важные цепи. Аналогично тому, как проводились исследования на старых моделях.
2. **Train-абляция:** Полноценно обучить модель с отключёнными физ. блоками → сравнить с полной моделью.

**Реализация:** Через конфигурацию/уставки (флаги типа `disable_division=True`, `disable_multiplication=True`, `disable_angle_gate=True`), а не через ручное переписывание кода.

## Q7: Улучшенный LR scheduler {#q7}

**Проблема:**
- Текущий cosine scheduler иногда долго не снижает LR, когда модели уже пора.
- Иногда модели нужен бóльший коэффициент (из-за этого стартовый LR поднят до 1e-3).
- Замечено, что повторный запуск обучения (всплески LR) помогает модели выскакивать из зависаний.

**Варианты решения:**
- `CosineAnnealingWarmRestarts` — периодические всплески LR.
- `OneCycleLR` — агрессивный подъём и спуск.
- Ручной restart: стратегия «обучить N эпох → сбросить LR → обучить ещё» (как приходилось делать).

**Применимо к:** pretrain (сколько примеров из осциллограммы) и fine-tuning.

## Q13: Критерий выбора лучшей модели {#q13}

**Проблема:** На небольшой тестовой выборке модель может долго не обновлять «best» чекпоинт, хотя по loss/валидации явно улучшается. Увеличение выборки замедляет обучение.

**Варианты:**
- Скользящее среднее loss вместо single-epoch снапшота.
- Combo-критерий: `val_loss * α + (1 - F1_macro) * (1 - α)`.
- Периодическая расширенная валидация (каждые K эпох проверять на бóльшей части).
- Разделение: «best по loss» и «best по F1» — два отдельных чекпоинта.

## Q15: Корректная визуализация mode=zone {#q15}

**Что проверить:** При зонной разметке каждая точка осциллограммы покрыта разным количеством предсказаний (из-за скользящего окна):
- Первая точка — 1 предсказание.
- Далее нарастает: 2, 3, ...
- Максимум ≈ `10 / 0.5 + 2 = 22` предсказаний (при шаге = полпериода).

**Что должно быть:** Усреднение вероятности предсказания по количеству перекрывающихся окон в каждой точке. Иначе начало и конец осциллограммы будут иметь нерепрезентативные значения.

## Q16: Восстановление исходного сигнала из спектра {#q16}

**Цель:** Визуальная проверка — насколько спектральное представление (гармоники 1-9 + низшие) полно передаёт исходный сигнал.

**Алгоритм:**
1. Из спектральных параметров (модуль + фаза каждой гармоники, рассчитанных каждые полпериода) восстанавливаем мгновенные значения через обратное преобразование Фурье.
2. Строим 8 графиков на одном рисунке: IA, IB, IC, IN, UA, UB, UC, UN.
3. На каждом графике: **сплошная** линия — исходный сигнал; **пунктирная** — восстановленный.
4. Используем гармоники 1-9 + низшие (периоды 2, 4, 6, 10). Симметричные составляющие НЕ используются (они производные от фаз).

**Параметры, влияющие на восстановление:** частота дискретизации, частота сети, номера используемых гармоник, шаг пересчёта спектра.

## Q18: Разметка по формулам РНМ {#q18}

**Цель:** Создать эталонную аналитическую разметку по формулам реле направления мощности (РНМ), которая будет использована как «первый учитель» при fine-tuning (первый этап файтюнинга для стабилизации модели).

**Связь с прошлым:** В экспериментах 2.6 делали разметку по формуле для ОЗЗ. Здесь идея аналогичная, но для более широкого класса событий.

### Приложение 1: Формула РНМ {#q18-приложение-1-формула-рнм}

Все расчёты ведутся в комплексных числах.

#### Шаг 1: Угол между напряжениями прямой последовательности

Рассчитать угол $\Delta\varphi$ между напряжениями прямой последовательности в текущей точке (СШ1) и 10 периодов назад (СШ2). Привести к диапазону $[-180°, 180°)$.

**Условие:** Расчёт производить при $|U_1^{СШ1}| > 0.05 \cdot |U_1^{СШ2}|$, иначе $\Delta\varphi = 0$.

#### Шаг 2: Откорректированные вектора напряжения

$$\dot{U}_{корр}^{СШ1} = \dot{U}_1^{СШ1} + k \cdot \dot{U}_1^{СШ2} \cdot e^{j \frac{\Delta\varphi}{2}}$$

где $k = 0.1$ (уставка, задаётся через ПО).

Упрощённый порядок расчёта:
- К углу вектора $\dot{U}_1^{СШ2}$ прибавить $\frac{\Delta\varphi}{2}$.
- Вычислить промежуточный вектор (нужны Re и Im составляющие).
- Суммировать с $\dot{U}_1^{СШ1}$ (умножив промежуточный на $k$).
- Вычислить угол результирующего вектора (модуль не нужен).

#### Шаг 3: Параметры РНМ

Определить: **модуль тока** и **угол** между откорректированным вектором напряжения и током.

**Корректировка угла максимальной чувствительности:**

$$\varphi_{коррект}^{СШ1} = \frac{\varphi(\dot{I}_1^{СШ1}, \dot{U}_1^{СШ1})}{2}$$

Диапазон: $-10° \le \varphi_{коррект}^{СШ1} \le 10°$ (за пределами — граничное).

Результирующий угол: $\varphi_{мч\_корр} = \varphi_{мч} + \varphi_{коррект}^{СШ1}$, где $\varphi_{мч} = -45°$ (уставка).

**Адаптивность зоны (α положительная):**

$$\alpha = \frac{1}{4 \cdot \sqrt{1 + T}}$$

где $T = \frac{I_{КЗ}}{I_д}$, $I_{КЗ} = 0.2$ о.е. (уставка), $I_д$ — действующее значение тока прямой последовательности на текущей выборке. $\alpha$ в радианах, ограничение: $\alpha \le 10°$.

**Адаптивность зоны (α негативная):** При активации накладки:

$$\alpha = \frac{2}{4 \cdot \sqrt{1 + T}}$$

Ограничение: $\alpha \le 15°$.

**Адаптивная уставка по модулю тока:**

$$I_{уст\_адап} = I_{отст} \cdot I_{1,уВ}^{t - t_{уст}}$$

где $I_{отст} = 0.8$ (коэффициент отстройки), $I_{1,уВ}^{t-t_{уст}}$ — ток прямой последовательности 10 периодов назад.

Ограничение: $I_{уст\_адап} \ge I_{min}$, где $I_{min} = 0.005 / 20 = 0.00025$ о.е.

**Условия блокировки адаптивной уставки (сброс до $I_{min}$):**
1. Если $|I_1^t| < I_{min}$ → блокировка на **10 периодов** (скользящее окно).
2. Если $|I_1^t| > I_{ном}$ (= 1/20) **ИЛИ** $|I_1^t| > 3 \cdot I_{1}^{t-t_{уст}}$ → блокировка на **50 периодов**.

#### Итоговое условие срабатывания РНМ

Выдаёт **1**, когда одновременно:
- Модуль РНМ > порог адаптивной уставки.
- Угол РНМ находится в зоне:
  - Не более: $\varphi_{мч\_корр} + 90° - \alpha_{пол}$
  - Не менее: $\varphi_{мч\_корр} - 90° + \alpha_{нег}$

**Важно:** Все углы должны быть приведены к единому диапазону, т.к. $-90°$ и $270°$ — одна точка.
