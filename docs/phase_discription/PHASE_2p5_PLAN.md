# План Фазы 2.5: Углубленная оптимизация данных и архитектур (Data-Centric Refinement)

> **Статус:** Активна
> **Цель:** Устранение ограничений Фазы 2, внедрение продвинутого пайплайна данных (Sliding Window, Downsampling), адаптация под сложную структуру сигналов (СШ/КЛ, фазные/линейные) и переход к Multi-Label классификации.

## 🎯 Основные цели фазы

1.  **Data Pipeline 2.0:** Переход от "одного окна с начала файла" к полноценному сэмплированию данных (Random Sliding Window) по всей длине осциллограммы.
2.  **Smart Features:** Реализация гибкой логики выбора сигналов (токи, напряжения шин/линий) и переход к физически интерпретируемым признакам (Модуль/Угол относительно фазы А базисного сигнала).
3.  **Optimization:** Сравнение стратегий прореживания данных (Stride vs Snapshot) для ускорения KAN.
4.  **Target Granularity:** Обучение на двух уровнях детализации: "4 базовых класса" и "Полный набор событий" (Multi-label).
5.  **Data Augmentation:** Применение аугментации временных рядов для борьбы с переобучением и увеличения вариативности обучающей выборки (добавление шума, временные сдвиги, масштабирование амплитуды в физически допустимых пределах).
6.  **Improved Reporting:** Внедрение расширенной системы логирования и статистики обучения с автоматической генерацией отчётов для последующего анализа и формирования научных выводов (с учётом длительного обучения моделей).
7.  **Class Balancing:** Внедрение механизмов учёта несбалансированности классов (нормальных режимов значительно больше, чем аварий) через взвешивание в функции потерь или метриках. Реализация иерархического подхода для обработки неизвестных подклассов аварий (если встречается ML_3_2_1_1, а в обучении был только ML_3_2_1 — классифицировать как родительский класс).
8.  **KAN Arithmetic Layers:** Исследование добавления специализированных слоёв для физически значимых арифметических операций (умножение для расчёта мощности, деление для сопротивления/проводимости) с корректной обработкой граничных случаев (деление на значения близкие к нулю). Цель — повышение интерпретируемости KAN через обучаемые физические преобразования.

### Этап 2: Стратегии формирования входа (Input Strategies)
*Здесь мы решаем проблему скорости KAN и физического смысла.*

1.  **Переход на Модуль/Угол:**
    *   [x] **Задача:** Заменить `Re/Im` на `Magnitude/Angle`.
    *   **Важно:** Фаза (Angle) сама по себе бесполезна, так как зависит от момента начала окна.
    *   [x] **Решение:** Считать разность фаз относительно **Напряжения Фазы А** (или линейного напряжения, или Тока А, если напряжения нет). $\phi_{rel} = \phi_{x} - \phi_{Ua}$. Это делает признак инвариантным к сдвигу во времени.

2.  **Реализация 3-х стратегий сэмплирования (для окна 200 мс / 320 точек):**
    *   [x] **Стратегия A (Dense/Raw):** Все 320 точек сырого сигнала (или 320 рассчитанных векторов RMS/Phase). *Самая тяжелая.*
    *   [x] **Стратегия B (Strided/Downsampled):** Расчет признаков (RMS/SymComp) с шагом (stride) **16 точек**.
        *   Итоговый размер: $320 / 16 = 20$ временных отсчетов.
        *   Это идеальный баланс для KAN (мало точек, много смысла).
    *   [x] **Стратегия C (Snapshot/Sparse):** Взятие только двух точек: $t_{current}$ и $t_{start}$ (начало окна, -200мс).
        *   Итоговый размер: 2 временных отсчета.
        *   Проверка гипотезы: "Достаточно знать, что было 'до' и что 'сейчас', чтобы понять тип аварии".

3.  **Сравнение подходов к данным:**
    *   **Задача:** Провести сравнительный анализ моделей, обученных на:
        *   **Вариант 1:** Все доступные аналоговые сигналы (как есть в файле).
        *   **Вариант 2:** Умный селектор (стандартизированные 6 каналов + восстановленные).
    *   Цель: Понять, дает ли избыточность сигналов прирост качества или вносит шум.

### Этап 3: Работа с целевыми метками (Targets)

1.  **Multi-Label подход:**
    *   [x] **Задача:** Изменить кодирование `y`. Вместо одного числа (класс 0..N) выдавать вектор бинарных флагов $[0, 1, 0, 1, ...]$.
    *   [x] Использовать `BCEWithLogitsLoss`.

2.  **Два уровня детализации:**
    *   [x] **Level 1 (Base):** 4 класса.
        *   `Normal` (Нет флагов)
        *   `ML_1` (Рабочая коммутация)
        *   `ML_2` (Аномалия)
        *   `ML_3` (Авария)
        *   *Примечание:* Это требует маппинга исходных `ML_...` колонок в эти 4 категории. И то есть сигнал `ML_1_1` относится к рабочей коммутации, важна только первая цифра.
    *   **Level 2 (Full):** Все уникальные флаги.

3.  **Взвешивание классов и метрики:**
    *   **Проблема:** Критическая несбалансированность датасета (Normal режимы ~70-80%, аварийные ML_3_* ~5-10%).
    *   **Задача:** Реализовать механизмы учёта несбалансированности:
        *   **Class Weights:** Рассчитать веса классов `w_i = N_total / (N_classes * N_i)` и внедрить в `BCEWithLogitsLoss(pos_weight=weights)`.
        *   **Balanced Sampling:** Рассмотреть `WeightedRandomSampler` для более частого сэмплирования редких классов.
        *   **Метрики:** Переход от общей Accuracy к **Balanced Accuracy**, **F1-macro**, **Precision/Recall per class**.
    *   **Код:** Найти существующие наработки по взвешиванию в старом коде обучения и адаптировать.

4.  **Иерархическая обработка неизвестных подклассов:**
    *   **Проблема:** В Test/Val может встретиться уникальный подкласс аварии (например, `ML_3_2_1_1`), которого не было в Train.
    *   **Решение:** Использовать иерархическую структуру меток:
        *   Если модель видит `ML_3_2_1_1`, но обучалась только на `ML_3_2_1` — считать его родительским классом.
        *   Реализовать функцию `map_to_parent_class(label, trained_labels)` для валидации.
    *   **Метрика:** Добавить Hierarchical Precision (учёт частичного совпадения на уровне родительских классов).

### Этап 4: Адаптация моделей

1.  **Обновление входных слоев:**
    *   Модели должны принимать динамическое количество каналов (в зависимости от файла конфигурации `input_channels`).
    *   Для KAN/MLP: Учет изменения размера входа при смене стратегии (20 точек vs 320 точек).

2.  **Обновление выходных слоёв (Heads):**
    *   Реализовать `MultiHead` или просто конфигурируемый выходной слой.
    *   Head 1: 4 нейрона (для Base классов).
    *   Head 2: N нейронов (для всех флагов).
    *   *Гипотеза:* Можно учить одну сеть с двумя головами одновременно (Multi-task learning).

3.  **KAN с арифметическими операциями (KAN Arithmetic Layers):**
    *   **Мотивация:** Органы РЗА работают с физическими величинами: мощность ($S = U \cdot I$), сопротивление ($Z = U / I$), проводимость ($Y = I / U$).
    *   **Задача:** Разработать специализированные слои для KAN:
        *   **MultiplicationLayer:** Поэлементное умножение двух входных векторов (для расчёта мощности).
        *   **DivisionLayer:** Деление с обработкой нулевых знаменателей:
            *   Вариант 1: Добавление $\epsilon = 10^{-6}$ к знаменателю.
            *   Вариант 2: Clipping знаменателя: `denominator = max(abs(denominator), eps)`.
            *   Вариант 3: Маскирование выхода при малых знаменателях.
        *   **Интеграция:** Вставить эти слои между KAN-блоками, чтобы сеть могла обучать физические преобразования.
    *   **Пример архитектуры:**
        ```
        Input (U, I) -> KAN_Block -> [U_feat, I_feat] -> MultiplicationLayer -> Power_feat -> KAN_Block -> Output
                                                       -> DivisionLayer -> Impedance_feat -> ...
        ```
    *   **Интерпретируемость:** После обучения можно визуализировать, какие комбинации $U$ и $I$ создаёт сеть, и сравнить с классическими формулами РЗА.
    *   **Код:** Реализовать в `osc_tools/ml/kan_conv/arithmetic_layers.py`.

## ⚠️ Риски
-   **Отсутствие фазы А:** Для расчета относительного угла нужен опорный вектор. Если $U_A$ и $I_A$ отсутствуют или зашумлены, относительный угол "поплывет". *Митигация:* Использовать фазу B или C, если А недоступна, или использовать приращения ($\Delta \phi$).
-   **Несбалансированность линейных/фазных:** Модель, обученная на файлах с фазными напряжениями, может плохо работать на файлах только с линейными. *Митигация:* Обучать разные модели или использовать Dropout каналов (случайно занулять фазные напряжения в обучении, заставляя сеть учить линейные).
